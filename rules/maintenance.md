# 維護協議：怎麼安全地更新這套制度

這套檔案是活的，但改壞的成本很高（每個 session 都會載入 CLAUDE.md）。照這份協議改。

## 0. 不管改什麼，先備份

```bash
cp {要改的檔} ~/.claude/rules/backups/{檔名}.$(date +%Y%m%d-%H%M%S).bak
```
backups/ 超過 10 個檔時，刪最舊的、只留每檔最新 2 份。

備份後、動筆前，先讀 [diagnosis.md](diagnosis.md)：你的改動如果讓三大漏洞的任何一個更容易復發，就不要改。

## 1. 權限分級

**可自行改（改完跟使用者講一句即可）**：
- 各 rules/*.md 的「教訓」小節新增條目（格式見第 2 節）
- prompts.md 新增模板或改善填空說明
- 修正確定過時的事實（例：某指令已不存在——但要先驗證真的不存在，再改）
- letter.md 的交接區

**改之前要先問使用者（一句話說明想改什麼、為什麼）**：
- CLAUDE.md 的硬規則區（增、刪、放寬任何一條）
- delegation.md 的升降級路徑與「驗證不自驗」原則
- judgment.md 的任何判準門檻（例：把「兩輪」改成「三輪」）
- 刪除任何整個小節或整個檔案

**永遠不准做**：
- 因為「這條規則讓這次任務變麻煩」就刪規則。規則擋到你，通常代表它在工作。
  真心認為規則錯了 → 走「先問使用者」路徑，附上具體案例。
- 在 CLAUDE.md 塞長內容。CLAUDE.md 只放硬規則和路由，內容一律進 rules/ 檔。

## 2. 踩坑後教訓寫回哪裡、什麼格式

判斷歸屬：跟**派工/模型選擇**有關 → delegation.md；跟**判斷時機**有關 → judgment.md；
跟**這個環境/使用者**有關 → letter.md 交接區；跟**特定專案**有關 → 該專案自己的 CLAUDE.md（不要寫進全域）。

格式（加在對應檔案末尾的「## 教訓」小節，沒有就建一個）：

```
- [2026-07-05] 情況：{一句話}。錯誤做法：{做了什麼}。正確做法：{該做什麼}。
```

一條教訓 ≤3 行。同一個坑第二次出現 → 它不該只是教訓，把它升級成該檔正文的規則（走第 1 節對應權限）。

## 3. 精簡門檻（防止制度膨脹成沒人讀的長文）

| 檔案 | 上限 | 超過就 |
|---|---|---|
| CLAUDE.md | 40 行 | 把細節下放到 rules/ 檔，只留路由 |
| 各 rules/*.md | 250 行 | 開一個精簡 pass：合併重複、刪一年內沒用到的教訓 |
| 教訓小節 | 每檔 10 條 | 最舊的合併或升級成正文規則 |

精簡也是「改檔」，走第 0 節備份 + 第 1 節權限。

## 4. 改完必做的驗證（不可跳過）

1. read-back：派 fresh-context `haiku` agent 用 [prompts.md](prompts.md) 末尾的 read-back 格式讀改過的檔，
   確認它讀出來的意思 = 你想表達的意思。
2. 檢查路由完整性：CLAUDE.md 路由表裡的每個檔案路徑真實存在（`ls ~/.claude/rules/`）。
3. 跟使用者回報：改了哪個檔、哪一條、為什麼，一句話。

## 教訓

（尚無。第一條會由未來 session 依第 2 節格式寫入。）
