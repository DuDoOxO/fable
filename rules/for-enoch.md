# 給 Enoch：這套制度怎麼用（人類使用說明）

這份寫給你，不是寫給模型。模型的規則在同目錄其他檔案，它們每個 session 會自動照做。

## 平常（9 成的情況）：什麼都不用做

一般模型（Sonnet/Opus/Haiku）開場自動載入 `~/.claude/CLAUDE.md`，會自己：
派 subagent 掃 repo、完成前驗證、給你「到哪裡看、預期看到什麼」的驗收指引。
如果它沒照做，打一句「**看 CLAUDE.md 硬規則 N**」就會回到軌道。

## 需要「Fable 等級判斷」的時候：打斜線指令

判斷力已經被寫成流程——一般模型 + 正確流程 ≈ 強模型的大部分表現。
四個流程做成了指令（在 `~/.claude/commands/`），任何專案都能直接打：

| 情況 | 打這個 |
|---|---|
| 困難或高風險任務開場（大功能、動資料庫、大重構） | `/hard 任務描述` ——先方案比較給你選、先定驗收條件、最後獨立驗收 |
| 想要第二雙眼睛看目前的方案 | `/second-opinion` ——派 fresh opus 對抗性挑錯 |
| 懷疑它說完成但沒真的好 | `/verify-done` ——派 fresh agent 獨立驗收，不自驗 |
| 鬼打牆、越修越糟 | `/stuck` ——強制停止重試，先驗證改動有生效、再往上游二分定位 |

搭配：整場硬仗開場先 `/model opus` + `/effort high`（結束調回來省額度）。
指令沒出現在自動補全時，直接把上表右欄的意思用一句話打出來也有同樣效果
（規則檔都在，模型讀得到）。模型把規則檔改壞了 → 「照 maintenance.md 從 backups/ 恢復」。

## 制度補不了的三種題（誠實條款）

**品味題**（UX 好不好看、文案順不順）、**模糊需求**（你自己還沒想清楚要什麼）、
**大架構取捨**。這三種再強的流程也只能逼近，不能取代你。遇到時最有效的用法：

1. 要求「做 2–3 個版本讓我比」——把品味判斷留給你自己，這是模型做不到、你 10 秒就能做的事。
2. 需求還模糊時，先說「先不要寫 code，用文字跟我確認你理解的需求」，確認完再動手。

## 省額度的三個習慣（你常撞 rate limit，這些最有感）

1. **同一個 bug 它修兩次沒好，不要讓它修第三次**——打「換方法診斷」。第三次幾乎都是浪費。
2. **長 session 變鈍就 `/compact` 或開新 session**——新 session 便宜過拖著一車舊 context。
3. **開新專案的第一個 session 讓它建 CLAUDE.md**（規則裡有，它應該會主動做）——一次成本，之後每次都省。
