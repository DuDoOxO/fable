# 模型調度守則

原則一句話：**主對話是指揮官，負責拆解、派工、驗收、跟使用者說話；粗重活派 subagent。**
指揮官的 context 是全 session 最貴的資源——每塞進一個檔案全文，後面所有回合都要為它付費。

## 1. 指揮官不下場（何時必須派工）

符合任一條就派 subagent，不要自己做：

- 需要讀 **3 個以上檔案**才能回答的搜尋／理解類問題 → `Explore` agent
- **掃 repo**（找某功能在哪、找所有用到 X 的地方、盤點結構）→ `Explore` agent
- **查網頁／文件**（多頁瀏覽、比較資訊）→ `general-purpose` agent
- **批次改檔**（同樣的改動套到 4 個以上檔案）→ `general-purpose` agent
- **驗收自己的產出** → fresh-context agent（見第 6 節，這條沒有例外）

可以自己做、不必派工：改 1–3 個已知位置的檔案、跑一條指令看結果、回答不需查證的問題。
只有 1–2 個檔案要看就直接 Read，派工反而更貴（subagent 冷啟動要重新建立 context）。

## 2. 派工三件套（每個派工 prompt 必含）

1. **目標與動機**：要什麼、為什麼要（動機讓 subagent 在邊界情況能自己做對判斷）。
2. **驗收條件**：怎樣算做完，可檢查的具體條件（「測試通過」「回傳的每個位置都附 檔案:行號」）。
3. **回報格式**：明確規定回什麼、不回什麼（見第 4 節回報合約）。

缺一件就是壞派工。填空模板見 [prompts.md](prompts.md)。

## 3. 顯式指定 model（每次派工都要寫 model 參數）

派工只從 `haiku`、`sonnet`、`opus` 三個選。enum 裡若出現 `fable`，也不要指定它——
那是 2026-07 的一次性短期方案，使用者環境長期沒有這個模型，指定它可能直接報錯或落到未知行為。
表中沒涵蓋、拿不準難度的任務就用 `opus`。

| 任務 | model |
|---|---|
| 機械式搜尋、grep 型盤點、read-back 驗證、格式轉換 | `haiku` |
| 一般實作、探索理解、研究、審查、批次改檔 | `sonnet`（預設值，猶豫就選這個） |
| 跨檔架構設計、難 bug 的診斷、高風險判斷的第二意見 | `opus` |

**Effort**：Agent tool 沒有 per-agent effort 參數，不要在派工時嘗試設定。effort 是全域設定
（`~/.claude/settings.json` 的 `effortLevel`，目前 `medium`；使用者可用 `/effort` 調整）。
如果任務難到你想調 effort，正確動作是升級 model，不是動 effort。

## 4. 回報合約（寫進每個派工 prompt 的固定段落）

> 回報規則：只回結論與必要證據。提到程式碼位置一律用 `檔案路徑:行號`。
> 不要貼超過 10 行的程式碼或檔案內容。長產物（完整報告、大段程式碼、清單超過 20 項）
> 寫到檔案裡，回報只給檔案路徑加 3 行摘要。做不到或不確定的部分明講，不要編。

指揮官收到違約回報（貼了整個檔案、沒給行號）：這次先用，但下次派工把合約寫得更前面、更醒目。

## 5. 升降級路徑

- **haiku 錯一次 → 直接升 sonnet 重派**。不要跟 haiku 來回 debug，來回的成本超過升級。
- **sonnet 同一個子任務連錯兩次 → 升 opus**，且派工 prompt 必須附完整失敗軌跡：
  試了什麼、實際輸出／錯誤訊息、為什麼判定失敗。沒有失敗軌跡的升級 = opus 會重犯一樣的錯。
- **opus 解出模式後 → 降回 sonnet/haiku 批次套用**。例：opus 想通第一個檔案的正確改法，
  把「改法範本 + 一個已完成的例子」交給 sonnet 套到剩下的檔案。
- **同一件事最多重試兩輪**（不分模型層級累計）。兩輪後還不行 → 這不是執行力問題，
  是方向問題，回 [judgment.md](judgment.md)「方向錯了的訊號」處理，通常意味著要問使用者或換方法。

## 6. 驗證不自驗

寫產出的人不能當驗收的人——自驗只會確認自己的偏見。驗收一律派 **fresh-context** agent
（新派一個 Agent，不要 SendMessage 給剛做完事的那個），且不告訴它「這是我寫的」。

| 產出類型 | 驗法 |
|---|---|
| 檔案／文件 | read-back：派 `haiku` 讀檔，回答「這份檔案要求讀者做什麼」，答案跟原意不符就是寫得不清楚 |
| 程式碼 | 跑測試；沒測試就實跑（起 server、curl、開頁面）。環境跑不動時降級為「有條件完成」：必須明講「未驗證，風險點是 X」+ 給使用者一步驗證指引，不可宣稱已完成 |
| 高風險判斷（要刪東西、動 schema、對外發送） | 第二意見：派 `opus` 附上下文問「這個決定有什麼問題」；或生 2–3 個候選方案派 agent 評審選優 |

驗收 agent 的回報也走第 4 節合約：只回「通過／不通過 + 具體不符點（檔案:行號）」。
