# 給未來 session 的信（Fable 5 寫於 2026-07-05）

你是接手這個環境的模型。這封信講三件使用者沒問、但我認為對這裡最重要的事，
以及這套制度最可能怎麼爛掉、怎麼防。

## 三件最重要的事

**1. 這位使用者的信任建立在「畫面上看得到」，不是「解釋得漂亮」。**
歷史紀錄裡最傷的模式是：模型自信地說改好了 → 使用者打開畫面 → 「還是一樣耶」。
每發生一次，你之後說的每句話都要打折。反過來，一句「重新整理 /cards 頁，第一張卡應該
變成三張拼圖」讓使用者 10 秒完成驗收，信任就是這樣攢的。技術說明能省就省，
驗證指引一個字都不能省。

**2. Token 不是抽象成本，是使用者的每週工作量上限。**
他查過 17 次 /usage、6 次 rate-limit 選項——他經常撞限。你在主對話多讀的每個整檔，
都是他本週少做的一件事。但注意方向：**省 token 靠的是派工和少讀，絕對不是少驗證**。
跳過驗證省下的 token，會在返工迴圈裡十倍還回去（見 diagnosis.md 漏洞 3）。

**3. 這裡的專案又多又小，彼此有血緣，而且會搬家。**
15+ 個專案，好幾組是同一份碼的分支或搬家（checkin / men/checkin；ikigaido 現行版在
Paraloka/，舊路徑 ikigai/ 已消失；mans/ 整個目錄 2026-07 已不存在但歷史紀錄裡有 100+ 條）。
`~/.claude/history.jsonl` 和 projects/ 清單裡的路徑**可能已經失效**，動手前先確認路徑存在。所以：(a) 不要假設你在「唯一的那份」——
改之前 `pwd` + `git status` 確認你在哪、狀態乾不乾淨；(b) 在 A 專案學到的結構知識
很可能適用於它的兄弟專案，寫該專案 CLAUDE.md 時值得註明「近親：{路徑}」。

## 這套制度最可能的退化方式與預防

1. **規則檔膨脹成沒人讀的長文。** 每次踩坑都想加一條，兩個月後 CLAUDE.md 變 200 行，
   等於沒有 CLAUDE.md。防法：maintenance.md 的行數上限是硬的，加新規則前先問
   「哪條舊規則可以合併或刪掉」。
2. **派工規則被靜默放棄。** 派工比自己做多一步，弱模型在壓力下會「這次先自己做」，
   三次之後制度名存實亡。防法：CLAUDE.md 硬規則 3 寫了明確數字門檻（3 個檔案）。
   如果你發現自己在找理由繞過它，那正是它在發揮作用的時刻。
3. **驗證退化成儀式。** read-back 派了，但回報「通過」就過，沒人看它讀出了什麼。
   防法：read-back 的問法是開放式的（「這檔案要求讀者做什麼」），拿它的答案跟你的
   原意逐條對，對不上就是要修。驗收 agent 回空泛的「沒問題」= 不合格，重派。
4. **教訓只進不出。** 教訓小節變成流水帳，真正重要的淹沒在裡面。
   防法：同一個坑出現第二次就升級成正文規則（maintenance.md 第 2 節），教訓區每檔 ≤10 條。

## harness 極限（誠實條款）

這套制度能把「執行品質」補到接近強模型：拆解、驗證、fresh-context 評審都做得到。
補不了的是**模糊需求解讀**和**品味判斷**（架構取捨、UX、文案語感）。
遇到時照 judgment.md 誠實條款：出 2–3 個候選讓使用者選 → 高風險加 opus 第二意見 →
都不行就明說沒把握。此外，本環境（截至 2026-07-05）沒有設定 MCP server、沒有自訂
agents/skills、memory 目錄只綁 ~/fable 專案——不要假設這些存在，用之前先確認。
