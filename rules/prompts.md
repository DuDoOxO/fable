# 派工 prompt 模板

用法：複製對應模板，填 `{…}` 空格，作為 Agent tool 的 `prompt`。model 選擇見 [delegation.md](delegation.md) 第 3 節。
每份模板已內建三件套（目標與動機、驗收條件、回報格式），**不要刪掉回報規則段落**——刪了 subagent 就會把整個檔案倒回來。

共用回報規則（每份模板結尾都有，此處為母本）：

> 回報規則：只回結論。程式碼位置用 `檔案路徑:行號`。不要貼超過 10 行的原始內容。
> 長產物寫到 {輸出檔案路徑或 scratchpad}，回報給路徑 + 3 行摘要。
> 做不到或不確定的部分明講「未確認」，不要編造。

---

## 1. 搜尋（agent: Explore，model: haiku；找不到才升 sonnet）

```
在 {目錄/repo 路徑} 找出 {要找的東西，具體：函式名/功能/設定}。
動機：{為什麼要找，例如「接下來要改它的回傳格式」——這決定你該回報哪些細節}。
搜尋廣度：{medium / very thorough}。已知線索：{關鍵字、可能的檔名、慣例；沒有就寫「無」}。

驗收條件：每個找到的位置附 檔案:行號 與一句「它在做什麼」。
確定不存在也算合格答案，但要列出你搜過的目錄與關鍵字。

回報規則：只回結論。位置用 檔案:行號。不要貼超過 10 行原始內容。找不到就明講，不要猜。
```

## 2. 實作（agent: general-purpose，model: sonnet）

```
任務：{要做什麼，一句話}。
動機：{使用者要解決的問題——邊界情況照這個動機判斷}。
範圍：改 {明確檔案清單或目錄}；**不要**動 {排除範圍}。
現有慣例：{指出一個可模仿的既有範例，檔案:行號；沒有就寫「跟隨該目錄現有風格」}。

驗收條件（全部達成才算完成）：
1. {可觀察的行為改變，例：GET /api/cards 回傳含 images 陣列}
2. {驗證方式，例：npm test -- cards 通過 / curl 該端點確認格式}
3. 沒有留下 console.log、註解掉的舊碼、TODO。

回報規則：回「改了哪些檔（檔案:行號）+ 每檔一句改了什麼 + 驗證結果原文（測試輸出最後幾行）」。
驗證沒跑或沒過就照實說，不要說「應該可以」。
```

## 3. 重構（agent: general-purpose，model: sonnet；先跑一次測試確認基準）

```
重構目標：{例：把重複的 API 呼叫邏輯抽成共用函式}。
動機：{例：三個頁面各自複製了一份，每次改都會漏}。
不變式：**外部行為完全不變**。重構前先跑 {測試指令}，記下結果作為基準；
重構後再跑一次，結果必須一致。沒有測試的話，先寫下 {2–3 個手動驗證步驟} 並在前後各執行一次。
範圍：{檔案清單}。發現範圍外也該改的地方：記下來回報，**不要動**。

驗收條件：前後測試結果一致 + 重複程式碼已消除 + 沒有引入新的公開介面變更。

回報規則：回「基準測試結果 → 重構後測試結果（兩者原文最後幾行）+ 改動清單（檔案:行號）+ 範圍外發現」。
```

## 4. 研究（agent: general-purpose，model: sonnet）

```
問題：{要回答什麼，越具體越好，例：「Next.js 14 app router 下 X 套件還能用嗎，有無替代」}。
動機：{要做的決定——研究深度以「夠做這個決定」為準，不要寫論文}。
輸出：寫到 {scratchpad 或指定路徑}/research-{主題}.md，結構：結論（3 行內）→ 依據（每條附來源 URL）→ 沒查到的部分。

驗收條件：結論可直接拿來做決定；每個關鍵陳述有來源；查不到的明確標「未查證」。
禁止：把訓練記憶當查證結果。版本號、API 名稱、價格類資訊必須有本次查到的來源。

回報規則：回「結論 3 行 + 檔案路徑」。不要把整份報告貼回來。
```

## 5. 審查（agent: general-purpose，model: sonnet；高風險改動用 opus）

```
審查對象：{檔案清單或 diff 範圍}。這不是你寫的，請用挑錯的眼光讀。
背景：{這個改動想達成什麼}。
重點順序：1. 正確性 bug（會壞的）2. 驗收條件是否真的達成：{列出原驗收條件}
3. 安全問題（injection、權限、秘密外洩）4. 明顯的重複或死碼。
不用管：風格偏好、命名品味。

驗收條件：每個 finding 附 檔案:行號 + 一句為什麼是問題 + 嚴重度（會壞 / 該修 / 可改可不改）。
沒問題就回「通過」加上你檢查了哪些面向——空泛的「LGTM」不合格。

回報規則：只回 findings 清單或「通過 + 檢查面向」。不要複述程式碼在做什麼。
```

---

補充：驗收用的 read-back 派工（model: haiku）最簡格式：

```
讀 {檔案路徑}。不要參考其他任何資訊，回答：1. 這份檔案要求讀者在什麼情況下做什麼（條列）
2. 有沒有互相矛盾或看不懂的句子（附行號）。只回這兩題。
```
