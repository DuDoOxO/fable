為 $ARGUMENTS 繪製架構圖（未指定對象就畫當前專案）。讀者是產品導向的使用者：圖要一眼看懂，字要少而準。嚴格照四步執行：

## 第 1 步：偵察（主對話不下場掃 repo）

派 1 個 Explore agent（thorough），要它回答五題、每題附 檔案:行號：
(1) 進入點在哪（main/index/router）(2) 內部分成哪幾個主要模組、誰依賴誰
(3) 用了哪些外部服務與資料庫（含連線設定位置）(4) 怎麼部署（Dockerfile/k8s/Vercel…）
(5) 最重要的一條使用者流程，從進入點到資料落地經過哪些檔案。
專案有 CLAUDE.md 就先給 agent 讀，省探索成本。

## 第 2 步：畫圖（Mermaid，寫進該專案 docs/ARCHITECTURE.md）

最多三張，各有硬上限，超過就是你沒做取捨：
- **L1 系統上下文**：本系統一個方塊 + 使用者 + 外部服務/DB，≤8 節點
- **L2 模組圖**：內部主要模組與依賴方向，≤12 節點；utils/helpers/config 一律不畫
- **L3 關鍵流程**（可選）：第 1 步問題 (5) 那條流程的 sequenceDiagram，≤8 步

鐵律：
- 每個節點必須對應真實存在的目錄或檔案；檔尾附「節點 → 路徑」對照表。對不上的節點：刪掉，或標（推測）+ 虛線
- 每條邊都要有標籤（HTTP / gRPC / 讀寫 / 事件），箭頭方向全圖統一（= 呼叫方向或資料流向，擇一並註明）

## 第 3 步：兩層說明（贅字是缺陷）

- **圖下說明**：每張圖 ≤5 行，只寫圖上看不出來的事——為什麼這樣拆、已知限制、容易踩的坑。禁止複述圖面（「A 會呼叫 B」這種句子出現 = 重寫）
- **白話版**：≤5 句，不用術語，講清楚三件事：這系統給誰用、資料從哪裡來流到哪裡去、哪個部分壞了最致命

## 第 4 步：交付與驗證（缺一不可）

1. 對照表逐項 `ls` 確認路徑存在，不存在的回到第 2 步修圖
2. 另存 HTML 到同目錄 docs/architecture.html（與 md 併存、內容同步；內嵌 mermaid，script 用 https://cdn.jsdelivr.net/npm/mermaid@11/dist/mermaid.min.js，mermaid.initialize 後渲染）並 `open` 它讓使用者直接看圖；離線或失敗就明講「請把 docs/ARCHITECTURE.md 貼到 GitHub 或 mermaid.live 看圖」
3. 派 `haiku` read-back：只給它白話版那段，問「這個系統在做什麼、給誰用、哪裡壞了最嚴重」。它答不出來或答錯 = 白話版不合格，重寫再驗
